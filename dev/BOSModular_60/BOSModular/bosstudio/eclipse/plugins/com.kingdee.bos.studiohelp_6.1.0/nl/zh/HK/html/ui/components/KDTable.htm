Ø         功能列表

KDTable控件是一個大型的表格控件，繼承於JComponent。該控件收集了Ledger、KDGrid、KDSpread、JTable、Excel等功能需求。

KDTable支持數據顯示和編輯，而且可以靈活的更換顯示器（Renderer）和編輯器（Editor），通過更換Renderer和Editor可以達到不同的顯示效果和編輯效果。

KDTable支持樹型管理，提供更靈活的瀏覽方式。

KDTable支持數據融合和拆分，使得數據表現更為靈活。

KDTable支持數據分組以及基於分組的融合、統計和排序。分組統計可直接生成樹型結構。

KDTable內置腳本語言支持，並且支持腳本自定義擴展。

KDTable支持多種形式的導入導出方式，可以與Excel文件交換數據。

KDTable支持undo，redo功能。

²        如何指定行、列或單元的編輯器

KDTable的編輯器支持類型綁定和手動綁定兩種。

類型綁定即無須指定編輯器，KDTable將根據單元格的值類型，自動綁定到相應的編輯器上。目前內置支持的綁定有（String，JTextField）、（Boolean，JCheckBox）、（Date，KDDatePicker）、（其他類型，JTextField）。數據類型到編輯器的綁定是可擴充的。

手動綁定即用戶自己創建編輯器，並綁定到行、列或單元對像上。當編輯器綁定到行、列上時，將應用到行、列上的所有單元格。

手動綁定優先於類型綁定。

// 擴充類型綁定，指定表格中的整型都用指定編輯器編輯

table.putDefaultEditor(Integer.class, yourIntegerEditor);

 

// 下面的例子演示手動綁定JComboBox類型的編輯器到第三列上

 

// 創建一個JComboBox

JComboBox c = new JComboBox(new String [] {"", "hhh1", "hhh2" , "hhh3"});

// KDTable要求的編輯器必須實現KDTCellEditor接口

// KDTDefaultCellEditor是一個輔助類，可以幫助你將JComboBox轉化為KDTable支持的編輯器。

KDTDefaultCellEditor cellEditor = new KDTDefaultCellEditor(c);

// 將編輯器綁定到列對像上，此處同樣可綁定到行或單元對像上

table.getColumn(3).setEditor(cellEditor);

公式和宏

公式（Formula）和宏（Macro）通稱為腳本（Script）。在KDTable中，公式和宏的區分在於，公式是依附於單元格、列、行和整個表體的，而宏是依附於一個範圍（Range）的，範圍的實質是任意形狀的選區；公式的返回值決定了它所在的單元格的值，而宏是沒有返回值的可執行腳本，但是在宏中可以使用內置函數設置單元格的值以及格式等等。

公式是冒泡獲取的，冒泡的順序是單元格、列、行、表。如果單元格沒有自己的公式，那麼它首先試圖獲取列的默認公式，如果還是沒有，再獲取行的默認公式，直到頂層的表，如果最終還是不能獲取公式，那麼公式引擎認為該單元格沒有公式。

各個單元格之間的公式或者宏之間存在依賴關係，即一個單元格的公式或宏依賴於其它單元格的值或Style。這種依賴關係在一個KDTable內部運算的時候對用戶是透明的。在多表聯算（多個KDTable之間相互引用）的時候，外部系統需要管理這個依賴關係，參見下文「多表聯算」小節。

二次開發者需要注意的一點是：在KDTable內部，行列索引是從0開始的，使用API的時候也是如此，而在腳本語言中，用戶輸入的行列索引必須從1開始。

腳本相關接口都在com.kingdee.bos.ctrl.kdf.table.KDTScriptManager類中。本節介紹這些接口的使用方法，其中所使用的源代碼中假設已經產生了一個通過this.table引用的KDTable對象。

設置腳本源代碼

公式設置

公式都是以「=」開頭的，並且只有一句表達式。對於表、行、列、單元格都可以使用其setExpressions(String)方法設置其默認公式：

 

this.table.getRow(2).getCell(2).setExpressions(「=cell(1,2).getValue()」);

 

在有公式被設置的情況下，單元格原有值將被忽略。

宏設置

宏都是以字符「@」開頭的，可以有多條語句，可以編寫任意複雜的腳本。宏的設置方法與範圍（Range）相關：

 

KDTRange range = new KDTRange();

range.add(new RangeBlock(0, 2, 0, 2));

//設置一個區域內的宏

String macro = 「@cell.setValue(cell(1,1).getValue());cell.setForeColor(GREEN);cell.setBackColorHexString(\」#00ff00\」)」;

this.table.getScriptManager().setMacro(range, macro);

 

執行腳本

執行公式

公式可以被設置為自動運行，這樣的話每次調用setExpressions(String)的時候都會自動運行改動的公式。默認的情況下，公式被設置為自動運行。

設置公式不自動運行的方法如下：

 

table.getScriptManager().setRunFormulaAfterModify(false);

 

在批量添加公式的時候，應該將公式自動運行關閉，待添加完成以後一次運行。

運行指定單元格的公式的方法如下：

 

this.table.getScriptManager().runFormula(2, 2);

 

上面的這段代碼運行第2行第2個單元格的公式。該單元格的公式依賴的公式、依賴於該單元格的公式都會被自動重算。

除法公式除0導致錯誤問題
除法公式除0，默認單元格顯示：#DIV0 除數為零
可以通過KDTScriptManager. setDiv0DefaultValue(String)來設置要顯示默認錯誤信息

執行宏

宏可以被設置為自動運行，這樣的話每次調用setMacro(KDTRange,String)方法的時候，這段宏都會被自動執行，並且針對範圍中的每個單元格都執行一次。默認的情況下，宏被設置為自動運行。

設置宏不自動運行的方法如下：

 

table.getScriptManager().setRunMacroAfterModify(false);

 

在批量添加宏的時候，應該將宏自動運行關閉，待添加完成以後一次運行。

運行指定範圍內的宏的方法如下：

 

this.table.getScriptManager().runMacro(range);

 

上面的代碼將運行一個KDTRange對像range所指定的範圍對應的宏。這段宏依賴的宏、依賴於這段宏的宏都會被自動重新運行。

擴展腳本

KDTable的宏被設計為可擴展的，二次開發者可以添加自定義常量、函數，可以擴展變量解析接口來實現自定義變量。

自定義常量

KDTable腳本引擎內置了大量的常量供用戶使用（參見「語法參考」小節），除此以外，用戶還可以添加自定義常量，常量的值類型可以是任意類型。

添加自定義常量的時候，首先要創建一個KDTConstant對象，然後將它添加到腳本引擎的關鍵字表中：

 

KDTConstant key = new KDTConstant(「MyKeyword」,」MyValue」);

this.table.getScriptManager().putConstant(key);

 

需要為內置常量指定一個名稱和一個值，在KDTKeyword的構造函數中，值可以以Object類型的形式傳入。

除了值常量之外，用戶還可以指定公式常量，即一個對應於公式的常量，腳本引擎將自動計算其公式，並且用得到的值替換該常量。定義一個公式常量的方法如下：

 

String formula = 「=100 / 2」; // 你可以寫任何合法的公式

KDTConstant key = new KDTConstant(「MyKeyword」,formula,KDTConstant. TYPE_EXECUTABLE_SCRIPT);

this.table.getScriptManager().putConstant(key);

 

注意到KDTConstant構造函數的最後一個參數，這裡使用了常量KDTConstant.TYPE_EXECUTABLE_SCRIPT，這就表示常量的值是一段可執行腳本的運行結果，對應的是常量KDTConstant. TYPE_STATIC_OBJECT，它表示自定義常量的值是靜態值。

 

擴展變量解析器

有時候自定義常量還是不能滿足用戶的擴展需求，KDTable的腳本引擎支持二次開發者自定義變量解析器，並且掛接到腳本引擎中來使用。自定義變量解析器要實現接口IVariableParser。腳本引擎運行時，首先查找自定義常量表裡的常量，如果找不到指定的常量，將回調變量解析器接口。假定已經有了一個實現了該接口的類varParser，將它插入到腳本引擎的代碼如下：

 

this.table.getScriptManager().setVariableParser(varParser);

 

該接口的方法說明請參考相應的JavaDoc文檔。

 

添加函數提供者

腳本引擎使用反射機制來調用對象的方法，允許用戶插入對像作為方法提供者，這樣就可以在腳本中調用它的方法。

假設下面這段代碼中的myObj實現了consoleWrite(String)方法，添加這個方法提供者的代碼如下：

 

this.table.getScriptManager().addFunctionProvider(myObj);

 

那麼在腳本中用戶就可以直接調用這個方法consoleWrite()。比如：

 

consoleWrite(cell.getValue());

 

這段宏調用了這個方法，其參數是當前單元格的值。

腳本引擎內置了一個方法提供者隊列，執行時按照優先級從頭到尾搜索整個隊列，因此如果多個方法提供者存在同名方法，那麼處於隊列前面的方法將被使用，後面的會被忽略。

用戶也可以直接插入一個排好序的方法提供者數組，只需使用KDTScriptManager的方法addFunctionProviderList(ArrayList)即可。對應添加的方法，KDTScriptManager還提供了刪除函數提供者的方法：

 

public void removeFunctionProviderList(ArrayList list);

public boolean removeFunctionProvider(Object object);

 

接口說明請參見對應的JavaDoc。

 

多表聯算

KDTable腳本引擎支持同一進程中多個KDTable之間的相互引用。

實現這一功能的外部系統需要做兩件事：1）為每個表指定同一個TableRelations對象，腳本引擎使用這個對象維護依賴關係。2）外部系統需要管理在腳本中允許使用的表別名。

假設當前進程中存在兩個KDTable對像table和table2，指定TableRelations對象的步驟如下所示：

 

TableRelations relations = new TableRelations();

this.table.getScriptManager().setRelationsObject(relations);    this.table2.getScriptManager().setRelationsObject(relations);

此外，外部系統需要定義一個別名解析器。接口ITableAliasParser定義了表別名解析的方法，該方法返回表別名所表示的KDTable對像：

public KDTable parseAlias(String alias)

假設類Parser提供了一個示範的實現如下：

   public KDTable parseAlias(String alias)

    {

        if (alias.equalsIgnoreCase("t1"))

            return this.table;

        else if (alias.equalsIgnoreCase("t2"))

            return this.table2;

        else

            return null;

    }

使用如下代碼把Parser的對象parser插入到腳本引擎作為別名解析器：

this.table.getScriptManager().setAliasParser(parser);

this.table2.getScriptManager().setAliasParser(parser);

注意一點就是，如果一個KDTable對像中定義的腳本使用了一個別名，那麼一定需要對應的別名解析器來解析它，否則腳本執行將會出現錯誤。

上述兩個步驟完成以後，用戶就可以在腳本語言中使用t1和t2來引用這兩個不同的KDTable對像：

 

cell.setForeColorHexString("#00ff00");

cell.setValue(cell(1,1,"t2").getForeColor());

 

上面的腳本中，將對像table2的第1行第1個單元格的前景色改為綠色，然後將對像table的第1行第1個單元格的值設為table2的同一位置的單元格的前景色對象，在這裡是Color.GREEN對象。

事件機制

目前KDTable公式框架提供兩種粒度的事件，KDTScriptEvent事件在一批腳本運行之前和運行完成後被觸發，KDTCellScriptEvent事件在單個單元格的腳本運行完畢即觸發。調用一次KDTScriptManager的run() 方法，KDTScriptEvent只會被觸發兩次（開始和結束各一次），而KDTCellScriptEvent可能會被觸發多次，因為公式之間可能會存在依賴關係，一個單元格的公式的重算會引發其它單元格的公式重算。

通過如下方法將KDTScriptListener和KDTCellScriptListener加入到KDTable的事件列表中：

 

// l1實現了KDTScriptListener，l2實現了KDTCellScriptListener

this.table.getScriptManager.addKDTScriptListener(l1);

this.table.getScriptManager.addKDTCellScriptListener(l2);

 

關於接口和事件對象，請參見JavaDoc文檔。

格式化字符串語法

(I) %[....]     數值、日期等的Pattern表示字串

語法：

"%"["L|C|R"]["+|_|-"]["[ fill character]"]["{pattern}"][width]["." prec]type

 

1) 格式化字符串必須以%開頭

2) ["L|C|R"] 當要顯示的字符的個數少於[width]時，L標識左對齊，C標識居中對齊，R標識右對齊。默認的為右對齊

3) ["+|_|-"] +表示若是正數，也顯示符號。負數顯示」-「號。_表示無論正負，都不顯示符號。-表示只有負數才顯示符號。默認的為只有負數才顯示符號

4) [width] 寬度 規定了要顯示的字符的個數。如果要顯示的寬度大於[width]，則按實際的寬度來顯示；反之，則填補空格或按要求填補其它字符。

5) ["." prec] 精度 小數點後的位數。

6) type 類型:

           s - 字符串。如果指定了["." prec]，並且字符串的實際長度大於["." prec]，則從左到右截取精度指定數量的字符串，其餘的刪除。

            d -     十進制整數。指定["." prec]，若轉換後字符串小於給出的精度，在前邊填補0；大於精度，按實際長度顯示。

            o -     無符號八進制整數。指定["." prec]，若轉換後字符串小於給出的精度，在前邊填補0；大於精度，按實際長度顯示。

            x -     無符號十六進制整數。指定["." prec]，若轉換後字符串小於給出的精度，在前邊填補0；大於精度，按實際長度顯示。

            b -     無符號二進制數。指定["." prec]，若轉換後字符串小於給出的精度，在前邊填補0；大於精度，按實際長度顯示。

            f - 十進制浮點數。如果要轉換的是負值，則前面有一個負號。轉換後的數字，在小數點後面的數字的個數由["." prec]決定。如果沒有指定["." prec]，默認為2位精度。    

            n -     十進制浮點數。顯示千分符。其餘同f。

            e -     科學計數法。

            t -     日期時間。精度無效。

不要頻繁調用getRow、getColumn、getCell接口

因為上述接口都是通過動態創建IRow、IColumn、ICell對象的，頻繁調用不但影響性能而且會給gc帶來一定壓力。建議用臨時變量保存而不要頻繁調用。

 

示例: 建議使用代碼2， 而不是代碼1

 

代碼1（不推薦）：

table.getRow(0).getCell(0).setValue(「hello」);

table.getRow(0).getCell(1).setValue(「hello」);

table.getRow(0).getCell(2).setValue(「hello」);

。。。

 

代碼2（推薦）：

IRow row = table.getRow(0);

row.getCell(0).setValue(「hello」);

row.getCell(1).setValue(「hello」);

row.getCell(2).setValue(「hello」);

。。。

不要頻繁調用getStyleAttributes、getHeadStyleAttributes接口

KDTable、IRow、IColumn、ICell都提供了getStyleAttributes接口，該接口返回的StyleAttributes是通過內部對像動態組合而成，頻繁調用不但影響性能而且會給gc帶來一定壓力。建議用臨時變量保存而不要頻繁調用。另外，每修改StyleAttributes的一個屬性，內部都會觸發一個融合屬性的動作，該動作相對簡單的屬性賦值要耗時更多，建議使用示例中紅色的代碼來優化程序。

 

示例: 建議使用代碼2， 而不是代碼1

 

代碼1（不推薦）：

cell.getStyleAttributes().getInterior().setBackground(color);

cell.getStyleAttributes().getFont().setColor(color);

cell.getStyleAttributes().getFont().setName(strName);

cell.getStyleAttributes().getFont().setSize(size);

cell.getStyleAttributes().getProtection().setLocked(true);

。。。

 

代碼2（推薦）：

StyleAttributes sa = cell.getStyleAttributes();

// 避免每修改一個屬性都會發生融合動作

Object owner = sa.getOwner();

sa.setOwner(null);

sa.getInterior().setBackground(color);

sa.getFont().setColor(color);

sa.getFont().setName(strName);

sa.getFont().setSize(size);

sa.getProtection().setLocked(true);

。。。

// 將修改的所有屬性一併進行融合

sa.setOwner(owner);

sa.update();
