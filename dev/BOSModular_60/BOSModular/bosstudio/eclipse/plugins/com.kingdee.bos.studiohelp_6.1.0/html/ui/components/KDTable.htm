Ø         功能列表

KDTable控件是一个大型的表格控件，继承于JComponent。该控件收集了Ledger、KDGrid、KDSpread、JTable、Excel等功能需求。

KDTable支持数据显示和编辑，而且可以灵活的更换显示器（Renderer）和编辑器（Editor），通过更换Renderer和Editor可以达到不同的显示效果和编辑效果。

KDTable支持树型管理，提供更灵活的浏览方式。

KDTable支持数据融合和拆分，使得数据表现更为灵活。

KDTable支持数据分组以及基于分组的融合、统计和排序。分组统计可直接生成树型结构。

KDTable内置脚本语言支持，并且支持脚本自定义扩展。

KDTable支持多种形式的导入导出方式，可以与Excel文件交换数据。

KDTable支持undo，redo功能。

²        如何指定行、列或单元的编辑器

KDTable的编辑器支持类型绑定和手动绑定两种。

类型绑定即无须指定编辑器，KDTable将根据单元格的值类型，自动绑定到相应的编辑器上。目前内置支持的绑定有（String，JTextField）、（Boolean，JCheckBox）、（Date，KDDatePicker）、（其他类型，JTextField）。数据类型到编辑器的绑定是可扩充的。

手动绑定即用户自己创建编辑器，并绑定到行、列或单元对象上。当编辑器绑定到行、列上时，将应用到行、列上的所有单元格。

手动绑定优先于类型绑定。

// 扩充类型绑定，指定表格中的整型都用指定编辑器编辑

table.putDefaultEditor(Integer.class, yourIntegerEditor);

 

// 下面的例子演示手动绑定JComboBox类型的编辑器到第三列上

 

// 创建一个JComboBox

JComboBox c = new JComboBox(new String [] {"", "hhh1", "hhh2" , "hhh3"});

// KDTable要求的编辑器必须实现KDTCellEditor接口

// KDTDefaultCellEditor是一个辅助类，可以帮助你将JComboBox转化为KDTable支持的编辑器。

KDTDefaultCellEditor cellEditor = new KDTDefaultCellEditor(c);

// 将编辑器绑定到列对象上，此处同样可绑定到行或单元对象上

table.getColumn(3).setEditor(cellEditor);

²        列宽调整模式

支持三种列宽调整模式：自适应列宽，手动调整和固定列宽。默认处于自适应列宽模式，系统将根据索引列显示内容的长度调整索引列宽度，使之能完全显示；手动调整模式下最终用户可以使用鼠标拖拉索引列的宽度；固定列宽模式下，列宽会保持不变，用户也不可调整，但是可以调用API设置其宽度：

 

       table.getIndexColumn().setWidth(100);// 设置列宽为100px

 

KDTIndexColumn类定义了三个常量：WIDTH_AUTO，WIDTH_MANUAL，WIDTH_FIXED，分别表示自适应模式，手动模式和固定模式，设置方法如下：

 

       table.getIndexColumn.setWidthAdjustMode(KDTIndexColumn.WIDTH_AUTO);

公式和宏

公式（Formula）和宏（Macro）通称为脚本（Script）。在KDTable中，公式和宏的区分在于，公式是依附于单元格、列、行和整个表体的，而宏是依附于一个范围（Range）的，范围的实质是任意形状的选区；公式的返回值决定了它所在的单元格的值，而宏是没有返回值的可执行脚本，但是在宏中可以使用内置函数设置单元格的值以及格式等等。

公式是冒泡获取的，冒泡的顺序是单元格、列、行、表。如果单元格没有自己的公式，那么它首先试图获取列的默认公式，如果还是没有，再获取行的默认公式，直到顶层的表，如果最终还是不能获取公式，那么公式引擎认为该单元格没有公式。

各个单元格之间的公式或者宏之间存在依赖关系，即一个单元格的公式或宏依赖于其它单元格的值或Style。这种依赖关系在一个KDTable内部运算的时候对用户是透明的。在多表联算（多个KDTable之间相互引用）的时候，外部系统需要管理这个依赖关系，参见下文“多表联算”小节。

二次开发者需要注意的一点是：在KDTable内部，行列索引是从0开始的，使用API的时候也是如此，而在脚本语言中，用户输入的行列索引必须从1开始。

脚本相关接口都在com.kingdee.bos.ctrl.kdf.table.KDTScriptManager类中。本节介绍这些接口的使用方法，其中所使用的源代码中假设已经产生了一个通过this.table引用的KDTable对象。

设置脚本源代码

公式设置

公式都是以“=”开头的，并且只有一句表达式。对于表、行、列、单元格都可以使用其setExpressions(String)方法设置其默认公式：

 

this.table.getRow(2).getCell(2).setExpressions(“=cell(1,2).getValue()”);

 

在有公式被设置的情况下，单元格原有值将被忽略。

宏设置

宏都是以字符“@”开头的，可以有多条语句，可以编写任意复杂的脚本。宏的设置方法与范围（Range）相关：

 

KDTRange range = new KDTRange();

range.add(new RangeBlock(0, 2, 0, 2));

//设置一个区域内的宏

String macro = “@cell.setValue(cell(1,1).getValue());cell.setForeColor(GREEN);cell.setBackColorHexString(\”#00ff00\”)”;

this.table.getScriptManager().setMacro(range, macro);

 

执行脚本

执行公式

公式可以被设置为自动运行，这样的话每次调用setExpressions(String)的时候都会自动运行改动的公式。默认的情况下，公式被设置为自动运行。

设置公式不自动运行的方法如下：

 

table.getScriptManager().setRunFormulaAfterModify(false);

 

在批量添加公式的时候，应该将公式自动运行关闭，待添加完成以后一次运行。

运行指定单元格的公式的方法如下：

 

this.table.getScriptManager().runFormula(2, 2);

 

上面的这段代码运行第2行第2个单元格的公式。该单元格的公式依赖的公式、依赖于该单元格的公式都会被自动重算。

除法公式除0导致错误问题
除法公式除0，默认单元格显示：#DIV0 除数为零
可以通过KDTScriptManager. setDiv0DefaultValue(String)来设置要显示默认错误信息

执行宏

宏可以被设置为自动运行，这样的话每次调用setMacro(KDTRange,String)方法的时候，这段宏都会被自动执行，并且针对范围中的每个单元格都执行一次。默认的情况下，宏被设置为自动运行。

设置宏不自动运行的方法如下：

 

table.getScriptManager().setRunMacroAfterModify(false);

 

在批量添加宏的时候，应该将宏自动运行关闭，待添加完成以后一次运行。

运行指定范围内的宏的方法如下：

 

this.table.getScriptManager().runMacro(range);

 

上面的代码将运行一个KDTRange对象range所指定的范围对应的宏。这段宏依赖的宏、依赖于这段宏的宏都会被自动重新运行。

扩展脚本

KDTable的宏被设计为可扩展的，二次开发者可以添加自定义常量、函数，可以扩展变量解析接口来实现自定义变量。

自定义常量

KDTable脚本引擎内置了大量的常量供用户使用（参见“语法参考”小节），除此以外，用户还可以添加自定义常量，常量的值类型可以是任意类型。

添加自定义常量的时候，首先要创建一个KDTConstant对象，然后将它添加到脚本引擎的关键字表中：

 

KDTConstant key = new KDTConstant(“MyKeyword”,”MyValue”);

this.table.getScriptManager().putConstant(key);

 

需要为内置常量指定一个名称和一个值，在KDTKeyword的构造函数中，值可以以Object类型的形式传入。

除了值常量之外，用户还可以指定公式常量，即一个对应于公式的常量，脚本引擎将自动计算其公式，并且用得到的值替换该常量。定义一个公式常量的方法如下：

 

String formula = “=100 / 2”; // 你可以写任何合法的公式

KDTConstant key = new KDTConstant(“MyKeyword”,formula,KDTConstant. TYPE_EXECUTABLE_SCRIPT);

this.table.getScriptManager().putConstant(key);

 

注意到KDTConstant构造函数的最后一个参数，这里使用了常量KDTConstant.TYPE_EXECUTABLE_SCRIPT，这就表示常量的值是一段可执行脚本的运行结果，对应的是常量KDTConstant. TYPE_STATIC_OBJECT，它表示自定义常量的值是静态值。

 

扩展变量解析器

有时候自定义常量还是不能满足用户的扩展需求，KDTable的脚本引擎支持二次开发者自定义变量解析器，并且挂接到脚本引擎中来使用。自定义变量解析器要实现接口IVariableParser。脚本引擎运行时，首先查找自定义常量表里的常量，如果找不到指定的常量，将回调变量解析器接口。假定已经有了一个实现了该接口的类varParser，将它插入到脚本引擎的代码如下：

 

this.table.getScriptManager().setVariableParser(varParser);

 

该接口的方法说明请参考相应的JavaDoc文档。

 

添加函数提供者

脚本引擎使用反射机制来调用对象的方法，允许用户插入对象作为方法提供者，这样就可以在脚本中调用它的方法。

假设下面这段代码中的myObj实现了consoleWrite(String)方法，添加这个方法提供者的代码如下：

 

this.table.getScriptManager().addFunctionProvider(myObj);

 

那么在脚本中用户就可以直接调用这个方法consoleWrite()。比如：

 

consoleWrite(cell.getValue());

 

这段宏调用了这个方法，其参数是当前单元格的值。

脚本引擎内置了一个方法提供者队列，执行时按照优先级从头到尾搜索整个队列，因此如果多个方法提供者存在同名方法，那么处于队列前面的方法将被使用，后面的会被忽略。

用户也可以直接插入一个排好序的方法提供者数组，只需使用KDTScriptManager的方法addFunctionProviderList(ArrayList)即可。对应添加的方法，KDTScriptManager还提供了删除函数提供者的方法：

 

public void removeFunctionProviderList(ArrayList list);

public boolean removeFunctionProvider(Object object);

 

接口说明请参见对应的JavaDoc。

 

多表联算

KDTable脚本引擎支持同一进程中多个KDTable之间的相互引用。

实现这一功能的外部系统需要做两件事：1）为每个表指定同一个TableRelations对象，脚本引擎使用这个对象维护依赖关系。2）外部系统需要管理在脚本中允许使用的表别名。

假设当前进程中存在两个KDTable对象table和table2，指定TableRelations对象的步骤如下所示：

 

TableRelations relations = new TableRelations();

this.table.getScriptManager().setRelationsObject(relations);    this.table2.getScriptManager().setRelationsObject(relations);

此外，外部系统需要定义一个别名解析器。接口ITableAliasParser定义了表别名解析的方法，该方法返回表别名所表示的KDTable对象：

public KDTable parseAlias(String alias)

假设类Parser提供了一个示范的实现如下：

   public KDTable parseAlias(String alias)

    {

        if (alias.equalsIgnoreCase("t1"))

            return this.table;

        else if (alias.equalsIgnoreCase("t2"))

            return this.table2;

        else

            return null;

    }

使用如下代码把Parser的对象parser插入到脚本引擎作为别名解析器：

this.table.getScriptManager().setAliasParser(parser);

this.table2.getScriptManager().setAliasParser(parser);

注意一点就是，如果一个KDTable对象中定义的脚本使用了一个别名，那么一定需要对应的别名解析器来解析它，否则脚本执行将会出现错误。

上述两个步骤完成以后，用户就可以在脚本语言中使用t1和t2来引用这两个不同的KDTable对象：

 

cell.setForeColorHexString("#00ff00");

cell.setValue(cell(1,1,"t2").getForeColor());

 

上面的脚本中，将对象table2的第1行第1个单元格的前景色改为绿色，然后将对象table的第1行第1个单元格的值设为table2的同一位置的单元格的前景色对象，在这里是Color.GREEN对象。

事件机制

目前KDTable公式框架提供两种粒度的事件，KDTScriptEvent事件在一批脚本运行之前和运行完成后被触发，KDTCellScriptEvent事件在单个单元格的脚本运行完毕即触发。调用一次KDTScriptManager的run() 方法，KDTScriptEvent只会被触发两次（开始和结束各一次），而KDTCellScriptEvent可能会被触发多次，因为公式之间可能会存在依赖关系，一个单元格的公式的重算会引发其它单元格的公式重算。

通过如下方法将KDTScriptListener和KDTCellScriptListener加入到KDTable的事件列表中：

 

// l1实现了KDTScriptListener，l2实现了KDTCellScriptListener

this.table.getScriptManager.addKDTScriptListener(l1);

this.table.getScriptManager.addKDTCellScriptListener(l2);

 

关于接口和事件对象，请参见JavaDoc文档。

(I) %[....]     数值、日期等的Pattern表示字串

语法：

"%"["L|C|R"]["+|_|-"]["[ fill character]"]["{pattern}"][width]["." prec]type

 

1) 格式化字符串必须以%开头

2) ["L|C|R"] 当要显示的字符的个数少于[width]时，L标识左对齐，C标识居中对齐，R标识右对齐。默认的为右对齐

3) ["+|_|-"] +表示若是正数，也显示符号。负数显示”-“号。_表示无论正负，都不显示符号。-表示只有负数才显示符号。默认的为只有负数才显示符号

4) [width] 宽度 规定了要显示的字符的个数。如果要显示的宽度大于[width]，则按实际的宽度来显示；反之，则填补空格或按要求填补其它字符。

5) ["." prec] 精度 小数点后的位数。

6) type 类型:

           s - 字符串。如果指定了["." prec]，并且字符串的实际长度大于["." prec]，则从左到右截取精度指定数量的字符串，其余的删除。

            d -     十进制整数。指定["." prec]，若转换后字符串小于给出的精度，在前边填补0；大于精度，按实际长度显示。

            o -     无符号八进制整数。指定["." prec]，若转换后字符串小于给出的精度，在前边填补0；大于精度，按实际长度显示。

            x -     无符号十六进制整数。指定["." prec]，若转换后字符串小于给出的精度，在前边填补0；大于精度，按实际长度显示。

            b -     无符号二进制数。指定["." prec]，若转换后字符串小于给出的精度，在前边填补0；大于精度，按实际长度显示。

            f - 十进制浮点数。如果要转换的是负值，则前面有一个负号。转换后的数字，在小数点后面的数字的个数由["." prec]决定。如果没有指定["." prec]，默认为2位精度。    

            n -     十进制浮点数。显示千分符。其余同f。

            e -     科学计数法。

            t -     日期时间。精度无效。

改善性能

批量操作时避免多余的界面绘制

KDTable有很多接口（例如：addRow()、 cell.setValue等）都会触发界面的重绘，会严重影响性能。在窗口第一次show（也即KDTable第一次paint）之前，KDTable会自动确保不会触发多次绘制，但窗口显示之后，需要开发人员支持，确保不会多次触发绘制。使用方法及示例代码如下：

// 使用方法：

 

// 设置table不重绘

table.setRefresh(false);

// 批量操作

。。。

 

// 设置重绘标志位并重绘

table.setRefresh(true);

table.repaint();

 

 

 

// 示例一（show之前的初始化，无需做多余的操作）：

 

// 创建table对象

KDTable table = new KDTable();

 

// 添加数据

IRow row;

For (int I = 0; I < 100; i++)

{

    row = table.addRow(i);

    for (int j = 0; j < 20; j++)

    {

        row.getCell().setValue(“cell” + I + j);

    }

}

 

 

// 示例二（点击某个按钮触发的操作--show之后的动作）：

 

// 删除当前所有行

table.removeRows();

 

// 设置table不重绘

table.setRefresh(false);

 

// 添加数据

IRow row;

For (int I = 0; I < 100; i++)

{

    row = table.addRow(i);

    for (int j = 0; j < 20; j++)

    {

        row.getCell().setValue(“cell” + I + j);

    }

}

 

// 设置重绘标志位并重绘

table.setRefresh(true);

table.repaint();

不要频繁调用getRow、getColumn、getCell接口

因为上述接口都是通过动态创建IRow、IColumn、ICell对象的，频繁调用不但影响性能而且会给gc带来一定压力。建议用临时变量保存而不要频繁调用。

 

示例: 建议使用代码2， 而不是代码1

 

代码1（不推荐）：

table.getRow(0).getCell(0).setValue(“hello”);

table.getRow(0).getCell(1).setValue(“hello”);

table.getRow(0).getCell(2).setValue(“hello”);

。。。

 

代码2（推荐）：

IRow row = table.getRow(0);

row.getCell(0).setValue(“hello”);

row.getCell(1).setValue(“hello”);

row.getCell(2).setValue(“hello”);

。。。

不要频繁调用getStyleAttributes、getHeadStyleAttributes接口

KDTable、IRow、IColumn、ICell都提供了getStyleAttributes接口，该接口返回的StyleAttributes是通过内部对象动态组合而成，频繁调用不但影响性能而且会给gc带来一定压力。建议用临时变量保存而不要频繁调用。另外，每修改StyleAttributes的一个属性，内部都会触发一个融合属性的动作，该动作相对简单的属性赋值要耗时更多，建议使用示例中红色的代码来优化程序。

 

示例: 建议使用代码2， 而不是代码1

 

代码1（不推荐）：

cell.getStyleAttributes().getInterior().setBackground(color);

cell.getStyleAttributes().getFont().setColor(color);

cell.getStyleAttributes().getFont().setName(strName);

cell.getStyleAttributes().getFont().setSize(size);

cell.getStyleAttributes().getProtection().setLocked(true);

。。。

 

代码2（推荐）：

StyleAttributes sa = cell.getStyleAttributes();

// 避免每修改一个属性都会发生融合动作

Object owner = sa.getOwner();

sa.setOwner(null);

sa.getInterior().setBackground(color);

sa.getFont().setColor(color);

sa.getFont().setName(strName);

sa.getFont().setSize(size);

sa.getProtection().setLocked(true);

。。。

// 将修改的所有属性一并进行融合

sa.setOwner(owner);

sa.update();
